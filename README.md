# VideoSave

VideoSave — демонстрационный Next.js-сервис, который проверяет публичные ссылки и показывает только допустимые способы сохранения контента. Проект намеренно **не** скачивает медиа со сторонних платформ, не обходит технические ограничения и не использует логин, DRM-bypass, CAPTCHA-bypass, paywall-bypass или доступ к приватным публикациям.

## Возможности

- современный адаптивный интерфейс на Next.js, React, TypeScript и Tailwind CSS;
- главная страница с проверкой ссылки и понятными статусами;
- страницы «Как это работает», «Правила», «DMCA», «Конфиденциальность» и «Контакты»;
- `POST /api/check-video` с строгой allowlist-проверкой доменов, ограничением запроса, sanitization и rate limiting;
- mock-разрешение только для контролируемого адреса `https://demo.videosave.local/allowed/creative-commons-clip`;
- официальный сценарий для известных социальных платформ: вместо извлечения файла пользователю показывается безопасная рекомендация использовать встроенную функцию платформы;
- обязательное подтверждение прав перед демонстрационным скачиванием.

## Запуск локально

Требования: Node.js 20.9+ и npm.

```bash
npm install
npm run dev
```

Откройте [http://localhost:3000](http://localhost:3000). Для проверки полного UI нажмите «Попробовать демо» под полем URL.

Дополнительные команды:

```bash
npm run typecheck
npm run build
npm run start
```

## API

### `POST /api/check-video`

Тело запроса:

```json
{ "url": "https://demo.videosave.local/allowed/creative-commons-clip" }
```

Успешный демонстрационный ответ соответствует контракту из задания: `status`, `platform`, метаданные, `thumbnail`, и массив `formats`. Для известных внешних платформ API вернёт `not_allowed` с официальным вариантом сохранения. Некорректный URL, неразрешённый домен и слишком частые запросы вернут `error`.

## Подключение реального официального источника

Перед добавлением нового провайдера необходимо:

1. Получить и соблюдать условия официального API/SDK источника.
2. Разрешать файл только когда провайдер **явно** подтвердил права и вернул собственную временную URL-ссылку на скачивание.
3. Не принимать ссылку на файл от клиента и не выполнять запросы к произвольным адресам.
4. Перед каждым исходящим запросом повторно проверять DNS-результат и блокировать private, loopback, link-local и reserved IP-адреса (защита от DNS rebinding / SSRF).
5. Использовать распределённый rate limiting (например, Vercel KV / Upstash) вместо in-memory limiter при масштабировании.
6. Обновить политики, контакты, сроки хранения и юридические реквизиты для вашей юрисдикции.

## Деплой на Vercel

1. Создайте Git-репозиторий и загрузите проект.
2. В Vercel выберите **Add New → Project** и импортируйте репозиторий.
3. Framework Preset: **Next.js**. Команды по умолчанию: `npm run build` и `npm run start`.
4. Нажмите **Deploy**.
5. Для production подключите централизованный rate limiter и замените адреса `@videosave.example` в страницах на реальные.

Проект не требует секретов в демонстрационном режиме.

## Ограничения безопасности

Текущий endpoint не делает исходящих запросов по пользовательским URL. Он принимает только HTTP(S), запрещает credentials/non-standard ports, локальные адреса и любые хосты вне allowlist. Встроенный ограничитель допускает максимум 12 проверок в минуту на технический идентификатор. В лог ошибок не попадают URL, IP-адреса и тела запросов.
